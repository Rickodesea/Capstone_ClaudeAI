Optimization Model — Mathematical Formulation

DAMO 699 Capstone · Group 3 · Multi-Tenant Cluster Memory Scheduling · Single-Objective LP

Terminology

Throughout this document the term job refers to a containerized workload unit submitted by a tenant for execution on the cluster. In Kubernetes, this corresponds to a pod — the smallest schedulable unit. We use job for generality, as the model is scheduler-agnostic and not specific to Kubernetes.

§0 — System Architecture

The optimization model is a single-objective LP. It receives real-time state from the cluster manager each scheduling round and outputs a placement assignment. SLA and fairness shape its inputs — they are not objectives inside the LP.

Cluster Manager → $U_n,\, A_n$ + $\bar{v}_n$ + $\bar{W}_t$ + $\hat{m}_j,\, \hat{p}_j^{\text{CPU}}$ → LP: place jobs, maximize $Z$ → $x_{jn}$ → Cluster Manager submits
§1 — Decision Variable
$$x_{jn} \in \{0,1\}$$

$x_{jn} = 1$ if job $j$ is assigned to node $n$; $x_{jn} = 0$ otherwise. This placement matrix is the sole output of the optimizer. Everything else in the model is either a fixed parameter, a constraint, or a feedback input.

§2 — Sets
SymbolDefinition
$J$All jobs currently in the pending queue at this scheduling round
$N$All nodes available in the cluster
$T$All tenants sharing the cluster
$J_t$Jobs belonging to tenant $t$, where $J_t \subseteq J$ and $\bigcup_{t \in T} J_t = J$
§3 — Parameters
Node Memory — Fixed
SymbolMeaning
$M_n$Physical RAM installed on node $n$
$\tau_n$Memory tax — fixed overhead consumed by the OS, kubelet, and monitoring agents. Not available to tenant jobs.
$M_n^{\text{avail}}$ Available memory — physical RAM minus the OS tax. This is the fixed upper bound for all tenant job memory on node $n$:
$$M_n^{\text{avail}} = M_n - \tau_n$$
Node CPU — Fixed
SymbolMeaning
$C_n$Total CPU cores on node $n$
Node Memory — Dynamic (updated each scheduling round)
SymbolSourceMeaning
$\bar{v}_n \in [0,1]$ cluster mgr Rolling SLA violation rate on node $n$ — fraction of the last $K$ scheduling rounds in which actual memory on node $n$ exceeded $M_n^{\text{avail}}$. Computed and passed by the cluster manager.
$$\bar{v}_n^{(t)} = \frac{1}{K}\sum_{k=0}^{K-1} \mathbb{1}\!\left[\text{overload on node } n \text{ at round } t - k\right]$$
$\bar{v}_n = 0$: no recent violations. $\bar{v}_n = 1$: every recent round was overloaded.
$K$ and $k$.  $K \in \mathbb{Z}^+$ is the rolling window length — the number of past scheduling rounds included in the violation rate. It is a configuration scalar (set before the simulation, not a decision variable). The summation index $k = 0, 1, \ldots, K-1$ iterates over those $K$ most recent rounds.
$U_n$ cluster mgr Currently used memory on node $n$, measured in real time. Provided as input each round. The model does not track this itself.
$A_n$ cluster mgr Currently used CPU on node $n$ — sum of $\hat{p}_j^{\text{CPU}}$ of all running jobs on node $n$. Provided as input each round.
$R_n^{\text{avail}}$ derived Available remaining memory — raw capacity remaining after subtracting current usage:
$$R_n^{\text{avail}} = M_n^{\text{avail}} - U_n$$
$R_n^{\text{eff}}$ derived Effective remaining memory — capacity offered to new jobs after applying the SLA violation scaling. The function $S_{f_n}$ converts $\bar{v}_n \in [0,1]$ and applies it as a percentage reduction:
$$R_n^{\text{eff}} = S_{f_n}\!\left(R_n^{\text{avail}},\, \bar{v}_n\right) = \max\!\left(0,\; R_n^{\text{avail}} \cdot (1 - \bar{v}_n)\right)$$
No violations ($\bar{v}_n = 0$): $R_n^{\text{eff}} = R_n^{\text{avail}}$ — full remaining capacity.
Rising violations: $R_n^{\text{eff}}$ shrinks — the node is offered less capacity to new jobs until violations subside.
All violations ($\bar{v}_n = 1$): $R_n^{\text{eff}} = 0$ — node is fully blocked for new admissions.
$R_n^{\text{eff}}$ is the right-hand side of constraint C2 in the optimizer.
$R_n^{\text{CPU}}$ derived Remaining CPU capacity — cores available after currently running jobs:
$$R_n^{\text{CPU}} = C_n - A_n$$
No SLA adjustment is applied. CPU overcommitment is tolerated at runtime via kernel throttling, so this is a hard admission guard only.
Fairness Feedback — Dynamic (from cluster manager)
SymbolSourceMeaning
$\bar{W}_t$ cluster mgr Average scheduling delay for tenant $t$ — mean time between job submission and job assignment, accumulated over all of tenant $t$'s completed jobs. Cluster manager records arrival and scheduling timestamps per job.
$\bar{W} = \frac{1}{|T|}\displaystyle\sum_{t \in T} \bar{W}_t$ derived Cluster-wide average scheduling delay across all tenants.
$\omega_t \in [1,2] \subset \mathbb{R}$ derived Tenant priority weight — scales the placement value of tenant $t$'s jobs in the objective. Tenants waiting longer than average receive higher weight, so their jobs contribute more to the objective, making the optimizer prefer placing them. The function $f_\omega$ maps the excess wait fraction into $[0,1]$ and shifts the result to $[1,2]$:
$$\omega_t = f_\omega\!\left(\bar{W}_t,\, \bar{W}\right) = 1 + \min\!\left(1,\;\max\!\left(0,\;\frac{\bar{W}_t - \bar{W}}{\max(1,\,\bar{W})}\right)\right)$$
$\omega_t = 1$ when $\bar{W}_t \leq \bar{W}$ — no penalty for tenants with short waits.
$\omega_t = 2$ when $\bar{W}_t \geq 2\bar{W}$ — maximum priority for severely underserved tenants.
No free parameter is required; the function saturates naturally at $2\times$.
Node Utilization Weight (Dynamic)
SymbolSourceMeaning
$u_n \in [1,2] \subset \mathbb{R}$ derived Node utilization weight — a weight derived from the percentage of available memory currently in use, applied directly in the objective to consolidate workloads onto busier nodes. The function $f_u$ maps the utilization fraction to $[1,2]$:
$$u_n = f_u\!\left(\frac{U_n}{M_n^{\text{avail}}}\right) = 1 + \min\!\left(1,\;\max\!\left(0,\;\frac{U_n}{\max(1,\, M_n^{\text{avail}})}\right)\right)$$
$u_n = 1$: node is idle — no consolidation preference.
$u_n = 2$: node is fully loaded — highest preference for new placements.
Constraint C2 still prevents any physically infeasible placement regardless of $u_n$.
Job Parameters (from prediction layer)
SymbolSourceMeaning
$\hat{m}_j$ prediction layer P95 predicted memory for job $j$ — the 95th percentile of the highest predicted memory spike over the job's expected lifetime. Used in the placement constraint C2 and the objective.
$\hat{p}_j^{\text{CPU}}$ prediction layer P90 predicted CPU peak for job $j$ — the 90th percentile of the job's predicted CPU demand. Used in the CPU capacity constraint C4. No SLA feedback applies to CPU; kernel throttling handles any runtime overrun.
$\hat{l}_j$ prediction layer Predicted job lifetime. Used internally by the prediction layer to determine the time horizon for $\hat{m}_j$ and $\hat{p}_j^{\text{CPU}}$. Not used directly in the LP.
§4 — Feedback Loops

SLA and fairness are not objectives inside the LP — they are observed outcomes tracked by the cluster manager and fed back as adaptive parameters that reshape the LP's inputs each round.

🔴 SLA Feedback → Node Capacity

When actual memory on node $n$ exceeds $M_n^{\text{avail}}$ at runtime, the cluster manager records one overload event. It computes the rolling violation rate $\bar{v}_n$ over the last $K$ rounds and passes it to the model.

The model derives $R_n^{\text{eff}} = S_{f_n}(R_n^{\text{avail}}, \bar{v}_n)$. As violations persist, $\bar{v}_n$ rises and $R_n^{\text{eff}}$ shrinks — the node is offered less capacity to new jobs. As violations subside, $\bar{v}_n$ falls and $R_n^{\text{eff}}$ recovers toward $R_n^{\text{avail}}$.

One overload event = one SLA violation. Violations are counted, not prevented, by this mechanism. The model reacts by becoming more conservative — it cannot retroactively fix an overload.
🟢 Fairness Feedback → Job Priority

The cluster manager records arrival and scheduling time for each completed job, computes per-job delay $d_j = s_j - a_j$, and accumulates average wait time $\bar{W}_t$ per tenant. This is passed to the model each round.

The model derives $\omega_t$ for each tenant. Tenants with above-average wait receive higher weight in the objective — their jobs contribute more to $Z$, so the optimizer naturally prefers placing them. Fairness improves as a side effect of weighted maximization.

Fairness as a KPI: Variance of $\bar{W}_t$ across tenants (or Gini coefficient) is reported as a simulation outcome metric — not minimized inside the LP.
§5 — Optimization Objective
Single Objective — Maximize Weighted Memory Utilization
$$\max \quad Z = \sum_{j \in J} \sum_{n \in N} \omega_{t(j)} \cdot \hat{m}_j \cdot u_n \cdot x_{jn}$$

where $t(j)$ denotes the tenant of job $j$, $\omega_{t(j)} \in [1,2] \subset \mathbb{R}$ is that tenant's priority weight, and $u_n \in [1,2] \subset \mathbb{R}$ is the node utilization weight for node $n$.

What this does: Places jobs onto nodes such that the total P95 predicted memory assigned is as large as possible, weighted by two multiplicative factors — both parameter-free functions bounded in $[1,2]$:
  • $\omega_{t(j)}$ — tenant fairness: a job from a long-waiting tenant gets a higher weight (up to $2\times$), so the solver naturally favors it.
  • $u_n$ — node utilization: a job placed on an already-loaded node contributes more, consolidating workloads onto fewer machines and leaving idle nodes available for power-down.
When all $\omega_t = 1$ (equal waits) and all $u_n = 1$ (idle nodes), this reduces to plain memory utilization maximization.
Why not maximize admission count? Placing many small jobs with low memory consumption could score well on job count but poorly on utilization. Maximizing $\hat{m}_j \cdot x_{jn}$ directly targets memory efficiency — the primary research objective.
§6 — Constraints
  • C1
    One node per job. Each job is assigned to at most one node per scheduling round:
    $$\sum_{n \in N} x_{jn} \leq 1 \quad \forall j \in J$$
    $\leq 1$ (not $= 1$) — a job may remain unscheduled if no node has sufficient remaining capacity. Unscheduled jobs return to the queue for the next round, accumulating delay that increases $\bar{W}_t$ and eventually raises their $\omega_t$.
  • C2
    Memory capacity per node. Total P95 predicted memory of jobs assigned to node $n$ may not exceed its effective remaining capacity:
    $$\sum_{j \in J} \hat{m}_j \cdot x_{jn} \leq R_n^{\text{eff}} \quad \forall n \in N$$
    $R_n^{\text{eff}} = S_{f_n}(R_n^{\text{avail}}, \bar{v}_n)$ incorporates the OS tax, current usage, and the SLA-driven scaling. This is a hard constraint — no job can be assigned if it would push P95 load past $R_n^{\text{eff}}$.
    Overcommitment lives here. Tenants declare memory (often inflated). The prediction layer produces $\hat{m}_j$ (P95, much lower than declared). The model admits a job if $\hat{m}_j$ fits within $R_n^{\text{eff}}$ — even if the declared amount would not. An SLA violation occurs if the 5% tail of the prediction materializes at runtime.
  • C3
    Binary domain:
    $$x_{jn} \in \{0,1\} \quad \forall j \in J,\; n \in N$$
    This makes the problem a Mixed-Integer Linear Program (MILP). For large instances, $x_{jn} \in [0,1]$ (LP relaxation with rounding) is an acceptable approximation for simulation scale.
  • C4
    CPU capacity per node (conditional — active only when ENABLE_CPU_CONSTRAINT = True). Total P90 predicted CPU peak of jobs assigned to node $n$ may not exceed its remaining CPU capacity:
    $$\sum_{j \in J} \hat{p}_j^{\text{CPU}} \cdot x_{jn} \leq R_n^{\text{CPU}} \quad \forall n \in N$$
    $R_n^{\text{CPU}} = C_n - A_n$ is the remaining CPU cores after currently running jobs. No SLA-driven scaling is applied.

    Why this is off by default. Unlike memory (physical pages are exclusive), CPU time is shared by the OS scheduler across all processes simultaneously — a node with 8 cores can run hundreds of threads without any single one being denied. CPU overcommitment is handled by kernel throttling (cgroups), which slows a job but never kills it. Treating cores as hard exclusive capacity is therefore too restrictive and would unnecessarily block placements. Set ENABLE_CPU_CONSTRAINT = True only for CPU-bound workloads where throttling is unacceptable.
§7 — Complete Model
$$\max \quad Z = \sum_{j \in J} \sum_{n \in N} \omega_{t(j)} \cdot \hat{m}_j \cdot u_n \cdot x_{jn}$$
Subject to
$$\text{C1:} \quad \sum_{n \in N} x_{jn} \leq 1 \quad \forall j \in J$$ $$\text{C2:} \quad \sum_{j \in J} \hat{m}_j \cdot x_{jn} \leq R_n^{\text{eff}} \quad \forall n \in N$$ $$\text{C3:} \quad x_{jn} \in \{0,1\} \quad \forall j \in J,\; n \in N$$ $$\text{C4 (if \texttt{ENABLE\_CPU\_CONSTRAINT}):} \quad \sum_{j \in J} \hat{p}_j^{\text{CPU}} \cdot x_{jn} \leq R_n^{\text{CPU}} \quad \forall n \in N$$
Where
$$M_n^{\text{avail}} = M_n - \tau_n \qquad R_n^{\text{avail}} = M_n^{\text{avail}} - U_n$$ $$R_n^{\text{eff}} = S_{f_n}\!\left(R_n^{\text{avail}},\, \bar{v}_n\right) = \max\!\left(0,\; R_n^{\text{avail}} \cdot (1 - \bar{v}_n)\right)$$ $$R_n^{\text{CPU}} = C_n - A_n$$ $$u_n = 1 + \min\!\left(1,\;\max\!\left(0,\;\frac{U_n}{\max(1,\, M_n^{\text{avail}})}\right)\right)$$ $$\omega_t = 1 + \min\!\left(1,\;\max\!\left(0,\;\frac{\bar{W}_t - \bar{W}}{\max(1,\, \bar{W})}\right)\right)$$
Inputs each round: $U_n$ (current node memory usage), $A_n$ (current node CPU usage), $\bar{v}_n$ (SLA violation rate per node), $\bar{W}_t$ (average wait per tenant), $\hat{m}_j$ (P95 predicted memory per job), $\hat{p}_j^{\text{CPU}}$ (P90 predicted CPU peak per job) — all from the cluster manager or prediction layer.

Configuration parameters: $K$ (violation rolling window length), $\tau_n$ (memory tax per node), ENABLE_CPU_CONSTRAINT (boolean; default False — C4 is omitted when disabled). No free tuning coefficients — all weighting functions are parameter-free and bounded in $[1,2]$.
