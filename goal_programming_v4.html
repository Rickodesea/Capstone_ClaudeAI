Optimization Model — Mathematical Formulation

DAMO 699 Capstone · Group 3 · Multi-Tenant Cluster Memory Scheduling · Single-Objective LP

Terminology

Throughout this document the term job refers to a containerized workload unit submitted by a tenant for execution on the cluster. In Kubernetes, this corresponds to a pod — the smallest schedulable unit. We use job for generality, as the model is scheduler-agnostic and not specific to Kubernetes.

§0 — System Architecture

The optimization model is a single-objective LP. It receives real-time state from the cluster manager each scheduling round and outputs a placement assignment. SLA and fairness shape its inputs — they are not objectives inside the LP.

Cluster Manager → $U_n$ + $\bar{v}_n$ + $\bar{W}_t$ → LP: place jobs, maximize $Z$ → $x_{jn}$ → Cluster Manager submits
§1 — Decision Variable
$$x_{jn} \in \{0,1\}$$

$x_{jn} = 1$ if job $j$ is assigned to node $n$; $x_{jn} = 0$ otherwise. This placement matrix is the sole output of the optimizer. Everything else in the model is either a fixed parameter, a constraint, or a feedback input.

§2 — Sets
SymbolDefinition
$\mathcal{J}$All jobs currently in the pending queue at this scheduling round
$\mathcal{N}$All nodes available in the cluster
$\mathcal{T}$All tenants sharing the cluster
$\mathcal{J}_t$Jobs belonging to tenant $t$, where $\mathcal{J}_t \subseteq \mathcal{J}$ and $\bigcup_{t \in \mathcal{T}} \mathcal{J}_t = \mathcal{J}$
§3 — Parameters
Node Memory — Fixed
SymbolMeaning
$M_n$Physical RAM installed on node $n$
$\tau_n$Memory tax — fixed overhead consumed by the OS, kubelet, and monitoring agents. Not available to tenant jobs.
Node Memory — Dynamic (updated each scheduling round)
SymbolSourceMeaning
$\bar{v}_n \in [0,1]$ cluster mgr Rolling SLA violation rate on node $n$ — fraction of the last $K$ scheduling rounds in which actual memory on node $n$ exceeded $M_n^{\text{eff}}$. Computed and passed by the cluster manager.
$$\bar{v}_n^{(t)} = \frac{1}{K}\sum_{k=0}^{K-1} \mathbb{1}\!\left[\text{overload on node } n \text{ at round } t - k\right]$$
$\bar{v}_n = 0$: no recent violations. $\bar{v}_n = 1$: every recent round was overloaded.
$\theta_n = \gamma \cdot \bar{v}_n$ derived Per-node memory threshold — a safety buffer derived from recent SLA violations. Increases with $\bar{v}_n$, directly reducing usable capacity on that node. $\gamma \in (0,1)$ is a tunable scaling coefficient controlling how aggressively violations reduce available capacity.
$$\theta_n = \gamma \cdot \bar{v}_n$$
$M_n^{\text{eff}}$ derived Effective usable memory — the hard ceiling for all tenant jobs on node $n$, after removing the OS tax and the SLA-driven threshold:
$$M_n^{\text{eff}} = M_n(1 - \theta_n) - \tau_n = M_n(1 - \gamma \cdot \bar{v}_n) - \tau_n$$
No violations ($\bar{v}_n = 0$): $M_n^{\text{eff}} = M_n - \tau_n$ — full usable capacity.
Rising violations: $\theta_n$ grows, $M_n^{\text{eff}}$ shrinks — the node is treated more conservatively each round until violations subside.
$U_n$ cluster mgr Currently used memory on node $n$, measured in real time. Provided as input each round. The model does not track this itself.
$R_n = M_n^{\text{eff}} - U_n$ derived Remaining available memory on node $n$ — the capacity available to newly assigned jobs this round.
Fairness Feedback — Dynamic (from cluster manager)
SymbolSourceMeaning
$\bar{W}_t$ cluster mgr Average scheduling delay for tenant $t$ — mean time between job submission and job assignment, accumulated over all of tenant $t$'s completed jobs. Cluster manager records arrival and scheduling timestamps per job.
$\bar{W} = \frac{1}{|\mathcal{T}|}\displaystyle\sum_{t \in \mathcal{T}} \bar{W}_t$ derived Cluster-wide average scheduling delay across all tenants.
$\omega_t$ derived Tenant priority weight — scales the placement value of tenant $t$'s jobs in the objective function. Tenants waiting longer than average receive higher weight, so their jobs contribute more to the objective, making the optimizer prefer placing them.
$$\omega_t = 1 + \alpha \cdot \max\!\left(0,\; \frac{\bar{W}_t - \bar{W}}{\bar{W}}\right)$$
$\omega_t \geq 1$ always — no tenant is penalized for having low wait. When all tenants have equal wait, $\omega_t = 1$ for all $t$ and the objective reduces to unweighted utilization.
What is $\alpha$?
$\alpha$ is a fairness responsiveness coefficient — it controls how strongly the model corrects wait-time imbalances between tenants. It is a tunable parameter set before the simulation.
$\alpha$ valueEffectExample: tenant waiting 50% longer than average
$\alpha = 0$Fairness weighting disabled. All tenants treated identically.$\omega_t = 1$
$\alpha = 1$Moderate correction. Weight scales linearly with excess wait.$\omega_t = 1.5$
$\alpha = 2$Aggressive correction. Underserved tenants strongly preferred.$\omega_t = 2.0$
A higher $\alpha$ makes the model correct fairness imbalances faster but may sacrifice some utilization efficiency to do so. Running simulations with different $\alpha$ values is a natural part of the evaluation.
Job Parameters (from prediction layer)
SymbolSourceMeaning
$\hat{m}_j$ prediction layer P95 predicted memory for job $j$ — the 95th percentile of the highest predicted memory spike over the job's expected lifetime. Used in the placement constraint and objective.
$\hat{l}_j$ prediction layer Predicted job lifetime. Used internally by the prediction layer to determine the time horizon over which $\hat{m}_j$ is estimated. Not used directly in the LP.
§4 — Feedback Loops

SLA and fairness are not objectives inside the LP — they are observed outcomes tracked by the cluster manager and fed back as adaptive parameters that reshape the LP's inputs each round.

🔴 SLA Feedback → Node Capacity

When actual memory on node $n$ exceeds $M_n^{\text{eff}}$ at runtime (actual usage > effective capacity), the cluster manager records one overload event. It computes the rolling violation rate $\bar{v}_n$ over the last $K$ rounds and passes it to the model.

The model derives $\theta_n = \gamma \cdot \bar{v}_n$ and reduces $M_n^{\text{eff}}$ accordingly. As violations persist, the node's effective capacity shrinks, making it harder to assign new jobs there. As violations subside, $\bar{v}_n$ falls, $\theta_n$ drops, and capacity recovers.

One overload event = one SLA violation. Violations are counted, not prevented, by this mechanism. The model reacts by becoming more conservative — it cannot retroactively fix an overload.
🟢 Fairness Feedback → Job Priority

The cluster manager records arrival and scheduling time for each completed job, computes per-job delay $d_j = s_j - a_j$, and accumulates average wait time $\bar{W}_t$ per tenant. This is passed to the model each round.

The model derives $\omega_t$ for each tenant. Tenants with above-average wait receive higher weight in the objective — their jobs contribute more to $Z$, so the optimizer naturally prefers placing them. Fairness improves as a side effect of weighted maximization.

Fairness as a KPI: Variance of $\bar{W}_t$ across tenants (or Gini coefficient) is reported as a simulation outcome metric — not minimized inside the LP.
§5 — Optimization Objective
Single Objective — Maximize Weighted Memory Utilization
$$\max \quad Z = \sum_{j \in \mathcal{J}} \sum_{n \in \mathcal{N}} \omega_{t(j)} \cdot \hat{m}_j \cdot x_{jn}$$

where $t(j)$ denotes the tenant of job $j$, and $\omega_{t(j)}$ is that tenant's current priority weight.

What this does: Places jobs onto nodes such that the total P95 predicted memory assigned is as large as possible, weighted by tenant priority. A job from a tenant with higher $\omega_t$ contributes more to $Z$ — the optimizer prefers it over an equally-sized job from a lower-weight tenant. When all $\omega_t = 1$, this is pure memory utilization maximization.
Why not maximize admission count? Placing many small jobs with low memory consumption could score well on job count but poorly on utilization. Maximizing $\hat{m}_j \cdot x_{jn}$ directly targets memory efficiency — the primary research objective.
§6 — Constraints
  • C1
    One node per job. Each job is assigned to at most one node per scheduling round:
    $$\sum_{n \in \mathcal{N}} x_{jn} \leq 1 \quad \forall j \in \mathcal{J}$$
    $\leq 1$ (not $= 1$) — a job may remain unscheduled if no node has sufficient remaining capacity. Unscheduled jobs return to the queue for the next round, accumulating delay that increases $\bar{W}_t$ and eventually raises their $\omega_t$.
  • C2
    Memory capacity per node. Total P95 predicted memory of jobs assigned to node $n$ may not exceed its remaining available capacity:
    $$\sum_{j \in \mathcal{J}} \hat{m}_j \cdot x_{jn} \leq R_n \quad \forall n \in \mathcal{N}$$
    $R_n = M_n^{\text{eff}} - U_n$ incorporates the OS tax, the SLA-driven threshold $\theta_n$, and current usage. This is a hard constraint — no job can be assigned if it would push P95 load past $R_n$.
    Overcommitment lives here. Tenants declare memory (often inflated). The prediction layer produces $\hat{m}_j$ (P95, much lower than declared). The model admits a job if $\hat{m}_j$ fits within $R_n$ — even if the declared amount would not. This is overcommitment: the model accepts more jobs than the declared values would allow, because it trusts the predictions over the declarations. An SLA violation (overload) occurs if the 5% tail of the prediction materializes at runtime.
  • C3
    Binary domain:
    $$x_{jn} \in \{0,1\} \quad \forall j \in \mathcal{J},\; n \in \mathcal{N}$$
    This makes the problem a Mixed-Integer Linear Program (MILP). For large instances, $x_{jn} \in [0,1]$ (LP relaxation with rounding) is an acceptable approximation for simulation scale.
§7 — Complete Model
$$\max \quad Z = \sum_{j \in \mathcal{J}} \sum_{n \in \mathcal{N}} \omega_{t(j)} \cdot \hat{m}_j \cdot x_{jn}$$
Subject to
$$\text{C1:} \quad \sum_{n \in \mathcal{N}} x_{jn} \leq 1 \quad \forall j \in \mathcal{J}$$ $$\text{C2:} \quad \sum_{j \in \mathcal{J}} \hat{m}_j \cdot x_{jn} \leq R_n \quad \forall n \in \mathcal{N}$$ $$\text{C3:} \quad x_{jn} \in \{0,1\} \quad \forall j \in \mathcal{J},\; n \in \mathcal{N}$$
Where
$$\theta_n = \gamma \cdot \bar{v}_n \qquad M_n^{\text{eff}} = M_n(1-\theta_n) - \tau_n \qquad R_n = M_n^{\text{eff}} - U_n$$ $$\omega_t = 1 + \alpha \cdot \max\!\left(0,\;\frac{\bar{W}_t - \bar{W}}{\bar{W}}\right)$$
Inputs each round: $U_n$ (current node usage), $\bar{v}_n$ (SLA violation rate per node), $\bar{W}_t$ (average wait per tenant), $\hat{m}_j$ (P95 predicted memory per job) — all from the cluster manager or prediction layer.

Tunable parameters: $\gamma$ (SLA threshold sensitivity), $\alpha$ (fairness responsiveness), $K$ (violation rolling window), $\tau_n$ (memory tax per node).
