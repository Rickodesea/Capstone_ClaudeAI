================================================================================
INTRODUCTION
================================================================================

We are building a real-time optimization model for scheduling jobs in a
multi-tenant compute cluster. Jobs are assigned to nodes at the moment
scheduling requests are received. A job refers to any containerized workload
unit — analogous to a Kubernetes pod — submitted by a tenant for execution on
the cluster. The model is scheduler-agnostic and not specific to Kubernetes.


================================================================================
§0  INPUTS TO OPTIMIZER
================================================================================

The cluster monitor provides the following inputs to the optimizer each
scheduling round, and receives the placement assignment x_jn in return:

  Cluster monitor inputs:
    U_mem_n         Currently used memory per node
    vbar_n          SLA violation rate per node — fraction of rounds in last K
                    window where actual memory exceeded M_cap_n
    omega_delay_t   Scheduling delay weight per tenant — computed over last K
                    rounds; >= 1, rises when tenant wait exceeds cluster average

  Prediction layer inputs:
    P_mem_j         Maximum predicted memory per pending job
    P_cpu_j         P95 predicted CPU peak per pending job

The optimizer maximizes Z (Placement Score) and outputs x_jn.

  [Cluster Monitor] ---> U_mem_n, vbar_n, omega_delay_t, P_mem_j, P_cpu_j
                    ---> [LP: maximize Z]
                    ---> x_jn
                    ---> [Cluster Monitor submits jobs to nodes]


================================================================================
§1  DECISION VARIABLE
================================================================================

  x_jn  in  {0,1}

  x_jn = 1  : job j is assigned to node n this round
  x_jn = 0  : job j is not assigned to node n

This placement matrix is the sole output of the optimizer.


================================================================================
§2  SETS
================================================================================

  J       All jobs currently in the pending queue this scheduling round
  N       All nodes available in the cluster
  T       All tenants sharing the cluster
  J_t     Jobs belonging to tenant t;
          J_t subseteq J,  and  union over t in T of J_t  =  J


================================================================================
§3  PARAMETERS
================================================================================

────────────────────────────────────────
§3.1 Node Limitations
(Physical node properties — fixed facts about the hardware)
────────────────────────────────────────

  M_n           Physical RAM installed on node n.

  C_n           Total CPU cores on node n.

────────────────────────────────────────
§3.2 Configuration
(Set by the cluster operator before the simulation; does not change at runtime)
────────────────────────────────────────

  M_tax_n       Memory tax — fixed overhead reserved for the OS, kubelet, and
                monitoring agents. Not available to tenant jobs.
                Based on Google cluster-usage traces v3: the local Borglet agent
                reserves approximately 5% of physical RAM for system use, so
                the effective usable ceiling for tasks is ~95% of M_n.

  M_theta_n     Memory safety threshold — fixed memory buffer reserved per node
                as protection against runtime memory spikes. Operator-configured.
                Default: M_theta_n = 0.10 * M_n  (10% of physical RAM).
                Together with M_tax_n, this defines the hard capacity ceiling
                M_cap_n available to tenant jobs.

  K             Rolling window length — number of past scheduling rounds used
                to compute both vbar_n and omega_delay_t. Configuration scalar;
                see Appendix A and Appendix C.

  sigma_consolid_n  Fixed node consolidation weight — static bias that steers
                placement toward lower-indexed nodes, keeping the cluster
                consolidated:

                  sigma_consolid_n = |N| - n     for n = 0, 1, ..., |N|-1

                Node 0 gets the highest weight; the last node gets 1.
                Constant — does not change between scheduling rounds.

                NOTE: A sequencing constraint (node n may only receive jobs
                if node n-1 is non-empty) would create a recursive dependency
                the MILP cannot resolve in a single solve call. The fixed weight
                achieves the same consolidation bias without constraint coupling.

────────────────────────────────────────
Cluster Monitor Inputs
(Provided to the optimizer at the start of each scheduling round)
────────────────────────────────────────

  U_mem_n       Currently used memory on node n. Measured in real time.
                Named U_mem_n (not U_n) to distinguish from future CPU usage
                metrics such as U_cpu_n.

  vbar_n        Rolling SLA violation rate on node n — fraction of the last
                K scheduling rounds in which actual memory on node n exceeded
                M_cap_n. Real-valued, vbar_n in [0, 1].

                  vbar_n = 0  : no recent violations
                  vbar_n = 1  : every recent round was overloaded

                See Appendix A for how vbar_n is computed.

  omega_delay_t Priority weight for job j — inherited from the tenant that
                owns job j. Computed over the last K scheduling rounds.
                omega_delay_t >= 1.

                  omega_delay_t = 1    : no extra weight (tenant wait at or below
                                         cluster-wide average)
                  omega_delay_t > 1    : additional weight; the solver increases
                                         preference for placing job j
                                         (real-valued, no maximum)

                See Appendix C for how omega_delay_t is computed.

────────────────────────────────────────
Prediction Layer Inputs
(Provided by the prediction layer for each pending job)
────────────────────────────────────────

  P_mem_j       Maximum predicted memory for job j — the worst-case peak
                memory usage predicted over the job's expected lifetime.
                Based on Google cluster-usage traces v3 `maximum_usage` field.
                Used in constraint C2 and the objective.

  P_cpu_j       P95 predicted CPU peak for job j — 95th percentile of
                CPU demand predicted over the job's expected lifetime.
                Based on `tail_cpu_usage_distribution` (index [4]) from
                Google cluster-usage traces v3.
                Used in constraint C4. No SLA feedback applies to CPU;
                kernel throttling handles runtime overrun.

  P_life_j      Predicted job lifetime. Used by the prediction layer to
                determine the time horizon for P_mem_j and P_cpu_j.
                Not used directly in the LP.

────────────────────────────────────────
Derived
(Computed by the optimizer from the above inputs each scheduling round)
────────────────────────────────────────

  M_cap_n       Node capacity for tenant jobs — physical RAM minus OS tax and
                safety threshold. This is the fixed ceiling for all tenant job
                memory on node n:

                  M_cap_n = M_n - M_tax_n - M_theta_n

  M_avail_n     Remaining available memory after subtracting current usage:

                  M_avail_n = M_cap_n - U_mem_n

  M_eff_n       Effective remaining memory — capacity offered to new jobs
                after SLA violation scaling:

                  M_eff_n = max(0, M_avail_n * (1 - vbar_n))

                  vbar_n = 0  :  M_eff_n = M_avail_n   (full remaining capacity)
                  vbar_n = 1  :  M_eff_n = 0            (node blocked)

                M_eff_n is the right-hand side of constraint C2.

  omega_utilize_n       Memory utilization weight — ratio of currently used memory to
                the physical maximum M_n, mapped to [1, 2]. Calculated against
                the maximum (physical) capacity even though jobs will never
                fully utilize M_n due to tax and threshold reservations.
                Applied in the objective to consolidate workloads onto busier
                nodes. Constraint C2 prevents infeasible placements regardless
                of omega_utilize_n.  See Appendix B for the formula.


================================================================================
§4  FEEDBACK LOOPS
================================================================================

SLA and fairness are not objectives inside the LP — they are observed outcomes
tracked by the cluster monitor and fed back as adaptive inputs each round.

SLA Feedback --> Node Capacity:
  When actual memory on node n exceeds M_cap_n at runtime, the cluster
  monitor records one overload event. It computes vbar_n over the last K rounds
  and passes it to the optimizer. As violations persist, vbar_n rises and
  M_eff_n shrinks — the node accepts fewer new jobs. As violations subside,
  M_eff_n recovers. One overload event = one SLA violation; the model reacts
  conservatively but cannot prevent an overload retroactively.

Fairness Feedback --> Job Priority:
  The cluster monitor tracks per-job wait times (submission to assignment),
  computes omega_delay_t per tenant over the last K rounds, and passes it to
  the optimizer. Jobs whose tenant is waiting above average receive
  omega_delay_t > 1, so the solver naturally prefers placing them. Fairness
  improves as a side effect of weighted maximization.

  Fairness KPI: variance of Wbar_t across tenants (or Gini coefficient) is
  reported as a simulation outcome metric — not minimized inside the LP.


================================================================================
§5  OPTIMIZATION OBJECTIVE
================================================================================

Maximize the Placement Score Z:

  max  Z  =  SUM{j in J}  SUM{n in N}
               omega_delay_j * P_mem_j * omega_utilize_n * sigma_consolid_n * x_jn

NOTE: omega_delay_j = omega_delay_t where t = t(j). The per-job subscript j is
used in the objective because the sum is over jobs. Appendix C defines
omega_delay_t (per-tenant); each job inherits it. They are mathematically
equivalent.

Each placed job contributes its maximum predicted memory P_mem_j, scaled by
three multiplicative weights:

  omega_delay_j    >= 1                  tenant delay weight (fairness; per-job form)
  omega_utilize_n          >= 1, in [1, 2]       memory utilization weight (consolidation)
  sigma_consolid_n in {1, ..., |N|}      fixed node bias (low-index preference)

Together these steer the solver to:
  - prefer jobs from long-waiting tenants (omega_delay_t)
  - consolidate onto memory-loaded nodes (omega_utilize_n)
  - fill lower-indexed nodes first (sigma_consolid_n)


================================================================================
§6  CONSTRAINTS
================================================================================

C1 — One node per job:

  SUM{n in N} x_jn  <=  1      forall j in J

  <= 1 (not = 1): a job may remain unscheduled if no node has room.
  Unscheduled jobs stay in the queue, accumulating wait time that raises
  their omega_delay_t in subsequent rounds.

C2 — Memory capacity per node:

  SUM{j in J} P_mem_j * x_jn  <=  M_eff_n      forall n in N

  Hard constraint. M_eff_n incorporates the OS tax, safety threshold, current
  usage, and SLA-driven scaling.

  OVERCOMMITMENT: The model uses P_mem_j (maximum predicted memory) rather
  than the declared request, admitting more jobs than a naive declared-memory
  scheduler would. An SLA violation occurs only if actual runtime memory
  exceeds even the predicted maximum — indicating behavior outside the
  training distribution.

C3 — Binary domain:

  x_jn  in  {0,1}      forall j in J, n in N

  Makes the problem a Mixed-Integer Linear Program (MILP). For large
  instances, x_jn in [0,1] (LP relaxation with rounding) is an acceptable
  approximation for simulation scale.

C4 — Per-pair CPU fitment:

  x_jn = 0   if   P_cpu_j > C_n      forall j in J, n in N

  Blocks placing a job whose predicted CPU peak exceeds the node's total
  core count. CPU cores are time-shared — the kernel handles overcommit via
  throttling. This check blocks only the genuinely infeasible case where a
  job's peak demand exceeds the node's hardware ceiling.


================================================================================
§7  COMPLETE MODEL
================================================================================

  max  Z  =  SUM{j in J}  SUM{n in N}
               omega_delay_j * P_mem_j * omega_utilize_n * sigma_consolid_n * x_jn

Subject to:
  C1:  SUM{n in N} x_jn  <=  1                         forall j in J
  C2:  SUM{j in J} P_mem_j * x_jn  <=  M_eff_n         forall n in N
  C3:  x_jn  in  {0,1}                                 forall j in J, n in N
  C4:  x_jn = 0  if  P_cpu_j > C_n                     forall j in J, n in N

Where:
  (quantities derived by the optimizer from inputs and configuration)

  M_cap_n          =  M_n  -  M_tax_n  -  M_theta_n
  M_avail_n        =  M_cap_n  -  U_mem_n
  M_eff_n          =  max(0,  M_avail_n  *  (1 - vbar_n))
  sigma_consolid_n =  |N| - n,   for n = 0, 1, ..., |N|-1

Inputs per round:
  U_mem_n           current node memory usage                [monitor]
  vbar_n            SLA violation rate per node (K-window)   [monitor]
  omega_delay_t     per-tenant delay weight (K-window);       [monitor]
                    used as omega_delay_j = omega_delay_t where t=t(j)
  P_mem_j           Maximum predicted memory per job         [prediction]
  P_cpu_j           P95 predicted CPU peak per job           [prediction]

Configuration parameters (constant, not decision variables):
  K                 rolling window length                    [config]
  M_tax_n           memory tax per node                      [config]
  M_theta_n         memory safety threshold per node         [config]
  sigma_consolid_n  fixed node consolidation weight          [config]

APPENDIX
================================================================================

────────────────────────────────────────
A.  How vbar_n is Computed
────────────────────────────────────────

Additional symbols used in this calculation:

  K             Rolling window length — number of past scheduling rounds
                included in the violation rate. Configuration scalar.

  k             Summation index, k = 0, 1, ..., K-1.
                k=0 is the most recent round; k=K-1 is the oldest.

  1[condition]  Indicator function: 1 if condition is true, 0 otherwise.

Formula (at scheduling round r):

  vbar_n = (1/K) * SUM{k=0 to K-1} ( 1[ U_mem_n > M_cap_n at round r-k ] )

  vbar_n is the fraction of the last K rounds in which actual memory on
  node n exceeded M_cap_n (the capacity ceiling after OS tax and safety
  threshold). Passed to the optimizer as an input each round.

────────────────────────────────────────
B.  How omega_utilize_n is Computed
────────────────────────────────────────

  omega_utilize_n = 1  +  clamp( U_mem_n / max(1, M_cap_n),  0,  1 )

The denominator is M_cap_n (schedulable capacity after OS tax and safety
threshold), not the raw physical M_n. This measures how loaded the node is
relative to the memory actually available for tenant jobs. When a node is
fully packed to M_cap_n, the weight reaches 2. Adding 1 shifts the range
from [0, 1] to [1, 2]:

  omega_utilize_n = 1  : node is idle (no consolidation boost)
  omega_utilize_n = 2  : node is fully packed to M_cap_n (maximum consolidation boost)
  1 < omega_utilize_n < 2  : partial load (real-valued, continuous range)

NOTE: The printed display metric (U_n / M_n) is separate from this weight
and uses physical M_n for display purposes only. No code change needed for
that display.

────────────────────────────────────────
C.  How omega_delay_t is Computed
────────────────────────────────────────

omega_delay_t [LaTeX: omega_t^{delay}] is the per-tenant scheduling delay
weight, defined here. The cluster monitor computes one weight per tenant and
passes it to the optimizer. Each job j inherits it as omega_delay_j = omega_delay_t
where t = t(j); omega_delay_j is the per-job form used in the objective.
Both Wbar_t and Wbar are computed over the last K scheduling rounds (same
rolling window as vbar_n), so the weight reflects recent fairness, not
all-time history.

Additional symbols:

  Wbar_t      Average scheduling delay for tenant t — mean time from job
              submission to job assignment, over the last K scheduling rounds.

  Wbar        Cluster-wide average scheduling delay over the last K rounds:

                Wbar = (1 / |T|)  *  SUM{t in T} Wbar_t

Formula:

  omega_delay_t  =  1  +  max(0,  (Wbar_t - Wbar) / max(1, Wbar))

  For job j owned by tenant t(j):
    omega_delay_j  =  omega_delay_t  where t = t(j)  (per-job form used in objective)

The weight is built from the excess wait above the cluster average:
  - The fractional excess (Wbar_t - Wbar) / max(1, Wbar) is 0 when the
    tenant is at or below average, and grows proportionally above that.
  - Adding 1 gives a minimum of 1 (no unfairness = no extra weight).
  - There is no stated maximum: a tenant waiting far above average can
    receive omega_delay_t >> 1, giving their jobs proportionally higher priority.

  omega_delay_t = 1    when Wbar_t <= Wbar    (no boost)
  omega_delay_t > 1    when Wbar_t > Wbar     (boost grows with excess wait)

  omega_delay_j = 1    when Wbar_{t(j)} <= Wbar    (no boost for job j)
  omega_delay_j > 1    when Wbar_{t(j)} > Wbar     (boost for job j, inherited from tenant)
