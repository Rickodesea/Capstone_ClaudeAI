LSTM Model Evaluation Report 
May 15th, 2026 
 
Abstract 
This report presents a systematic evaluation of Long Short-Term Memory (LSTM) neural network 
architectures for multi-step CPU and memory resource forecasting within a Google Cluster Trace 
environment. The study examines five experimental configurations across two primary forecasting 
strategies — tenant-aware generalization and workload behavior learning — assessing their 
suitability for proactive scheduling and SLA-aware resource allocation. The dataset used is the 
Google Cluster Trace v3 (Cell A, Week 1, May 2019), comprising over one million observation 
windows at 5-minute granularity across 794 unique instances. An Exploratory Data Analysis 
(EDA) was conducted to characterize dataset quality, workload composition, and instance lifetime 
distributions, confirming readiness for LSTM ingestion. Model performance is evaluated using 
MAE, RMSE, SMAPE, and R² metrics, and results are benchmarked against recent related 
literature. Findings demonstrate that workload behavior learning with dedicated single-output 
LSTM models for CPU and memory achieves superior forecasting accuracy, with R² values 
reaching 0.968 for CPU and 0.947 for memory at a 30-minute horizon. The selected final model 
configuration integrates 12-hour lookback windows, 5-minute granularity, and multi-step outputs 
transformed into optimizer-ready scheduling parameters.

Key Findings 
Dataset Scale & Cleanliness: Zero duplicates, and only 0.12% missing values — fully ready 
for LSTM ingestion. 
Workload Composition: Priority Tiers — dominated by Production (53.8%) and Free (45.6%) 
tiers; latency-sensitive jobs are a critical 6.9% minority. Terminal Type — dominated by Kill 
(97.1%) and zero Finish. Scheduling Class — latency-sensitive jobs represent 6.9% of rows. 
LSTM Readiness: All 794 instances exceed minimum sequence length; 97.3% of sub-
sequences usable after gap detection (~1M training pairs). Per-sequence normalisation is 
essential due to skew and distribution shift.

Alshboul et al. is the closest comparison because it uses Google Cluster data, LSTM, and CPU/memory prediction with MAE/RMSE/MSE. 
Their feature selection is more 'usage-history based' while ours is more 'EDA-driven + domain-aware'. 
How They Predict CPU and Memory 
• 
They create sequential workload windows from Google Cluster data. 
• 
Past workload values are fed into the LSTM. 
• 
The LSTM learns temporal workload patterns. 
• 
The model predicts future CPU demand and memory demand. 
Our Approach 
Our approach also uses Google Cluster trace data and LSTM-based time-series forecasting for CPU and memory prediction. However, the 
pipeline is designed more specifically for SLA-aware workload optimization and scheduler support. Instead of relying mainly on usage-history 
features, the proposed framework combines EDA-driven analysis, workload behavior learning, temporal sequence engineering, and optimizer-
oriented forecasting outputs. The predicted sequences are further transformed into optimization-ready parameters such as average usage, 
peak usage, variance, and percentile-based workload indicators to support proactive resource management and scheduling decision. 
 

Page 10 
Final Model Selection 
 
Among the evaluated architectures, the workload behavior learning approach with separate 
specialized LSTM models for CPU and memory forecasting using a 30-minute prediction 
horizon was selected as the final prediction framework for the proposed scheduling system. 
 
This configuration achieved the strongest balance between forecasting accuracy, workload 
behavior learning, and operational suitability for proactive cluster scheduling. Compared to tenant-
aware generalization, workload behavior learning is better aligned with the project objective of 
modeling temporal workload evolution within production cluster environments. Additionally, 
separating CPU and memory into dedicated models improved specialization and reduced 
interference between heterogeneous resource utilization patterns. The 30-minute forecasting 
horizon was selected over the 1-hour horizon because it provided slightly stronger predictive 
stability while remaining more appropriate for real-time scheduling and SLA-aware resource 
allocation decisions. 
 
The final framework therefore consists of: 
• 
Workload behavior learning split strategy 
• 
Separate LSTM models for CPU and memory 
• 
12-hour lookback windows 
• 
5-minute workload granularity 
• 
Multi-step forecasting outputs transformed into optimizer-ready scheduling parameters
